# 众包任务推荐强化学习实验报告

## 1. 项目目标与总体思路

本项目研究如何将 DQN 系列强化学习方法应用到众包任务推荐中。当 worker 进入平台时，系统从候选 project 中推荐一个任务。推荐策略需要同时考虑两类利益：

- 参与者利益：worker 能看到更相关、更感兴趣、更可能完成的任务。
- 请求者利益：任务能获得更多、更高质量的回答，尤其是临近截止的任务能被及时推荐。

因此，我们将问题建模为强化学习决策过程：

```text
state = 当前 worker 与候选 project 的匹配特征
action = 推荐候选列表中的一个 project
reward = 根据参与者目标或请求者目标计算反馈
next_state = 推进到下一条按时间排序的 worker 到达事件
```

整体流程分为五个部分：数据清洗与特征工程、强化学习环境设计、DQN 模型实现与训练、参数调优与多变体对比、baseline 与报告分析。

## 2. 成员 A：数据清洗与特征工程

成员 A 负责将原始众包数据整理成可供强化学习环境直接读取的 event 数据。

原始数据包括：

- `data/project/`：project 基本信息，包括类别、行业、开始时间、deadline、entry 数等。
- `data/entry/`：worker 参与 project 的历史记录。
- `data/project_list.csv`：project 列表。
- `data/worker_quality.csv`：worker 质量属性。

预处理脚本为 `data_processed/data_process.py`，输出目录为 `data_processed/`。主要输出文件如下：

| 文件                    | 内容                           |
| ----------------------- | ------------------------------ |
| `enhanced_train.pkl`  | 训练集 event，按时间前 70%     |
| `enhanced_val.pkl`    | 验证集 event，按时间 70% - 85% |
| `enhanced_test.pkl`   | 测试集 event，按时间后 15%     |
| `worker_episodes.pkl` | 按 worker_id 分组的完整序列    |
| `worker_features.pkl` | worker 静态特征                |
| `norm.json`           | 训练集归一化参数               |
| `category_maps.json`  | 类别、子类别、行业映射表       |
| `info.json`           | 数据集规模和特征维度信息       |

数据集规模如下：

| 数据集 | 样本数 |
| ------ | -----: |
| train  | 134475 |
| val    |  28816 |
| test   |  28817 |

单条样本结构如下：

```python
{
    "worker_id": int,
    "timestamp": datetime,
    "candidate_projects": List[int],
    "candidate_features": List[Dict],
    "positive_project": int,
    "positive_index": int,
    "worker_history_at_t": List[int],
}
```

其中 `positive_project` 表示 worker 在历史数据中实际选择的项目，`positive_index` 表示该项目在候选列表中的下标，是训练奖励和 hit_rate 评估的重要依据。

成员 A 构造了 13 维候选项目特征：

```text
worker_quality
worker_history_count
worker_active_days
project_category
project_sub_category
project_industry
project_entry_count
project_duration_days
category_match
industry_match
project_popularity
worker_category_count
remaining_days
```

其中 `worker_category_count` 表示当前时刻之前 worker 在该类别上的历史完成次数，体现 worker 的专项能力；`remaining_days` 表示 project 距离 deadline 的剩余天数，用于刻画任务紧迫度和过滤过期任务。

除类别特征、二值匹配特征和 `remaining_days` 外，连续特征使用训练集统计量做 z-score 标准化。计数类长尾特征先做 `log1p`，再进行标准化，避免大数值特征主导模型训练。

## 3. 成员 B：强化学习环境设计

成员 B 负责 `env/` 文件夹，实现众包任务推荐环境 `CrowdRecEnv`，将预处理后的数据封装为 DQN 可以交互的形式。本部分只描述环境中已经实现的接口、状态构造、动作合法性判断和奖励函数；模型网络、训练循环和调参不属于成员 B 的实现范围。

### 3.1 环境读取接口

环境默认读取 `data_processed/` 中的数据，提供以下接口：

```python
load_split("train")
load_split("val")
load_split("test")
load_all_splits()
load_worker_episodes()
```

其中 `load_worker_episodes()` 用于读取按 worker_id 组织的 episode 数据，便于后续扩展为 worker 序列级强化学习。

### 3.2 状态设计

环境 state 是一个字典：

```python
{
    "features": np.ndarray,      # shape = [20, 13]
    "action_mask": np.ndarray,   # shape = [20]
    "worker_id": np.ndarray,
    "timestamp_index": np.ndarray
}
```

`features` 是候选任务特征矩阵。每一行表示一个候选 project，每一列表示固定顺序的特征。当前新版数据中最大候选数为 20，特征维度为 13，因此：

```text
state["features"].shape = [20, 13]
action_dim = 20
```

如果某条样本候选数不足 20，环境会 padding 到固定长度，并通过 `action_mask` 屏蔽 padding 位置。

### 3.3 动作设计

动作定义为候选项目下标：

```text
action ∈ [0, 19]
```

例如 `action = 3` 表示推荐当前候选列表中的第 4 个 project。DQN 输出长度为 20 的 Q 值向量，每个位置对应一个候选任务。

### 3.4 过期任务过滤

根据大作业要求，系统应考虑任务创建时间、到期时间和 worker 到达时间。环境使用 `remaining_days` 过滤已经过期的候选任务：

```text
remaining_days <= 0 时，action_mask 对应位置设为 0
```

如果旧数据没有 `remaining_days`，环境会尝试使用 `deadline` 和当前样本 `timestamp` 判断；如果两者都不存在，则默认该候选任务可推荐。

### 3.5 奖励设计

环境支持四种奖励类型。

参与者目标：

```text
reward = 1, 如果 action == positive_index
reward = 0, 否则
```

该奖励衡量推荐是否命中 worker 历史真实选择的任务。

请求者质量目标：

```text
reward = worker_quality, 如果 action == positive_index
reward = 0, 否则
```

该奖励鼓励将任务推荐给质量更高的 worker。

请求者紧迫度目标：

```text
reward = worker_quality + urgency_weight * urgency
urgency = 1 / (1 + exp(remaining_days))
```

当任务越接近 deadline，`remaining_days` 越小，`urgency` 越大。

混合目标：

```text
reward = alpha * worker_reward + (1 - alpha) * requester_reward
```

该奖励用于同时考虑参与者和请求者收益。

### 3.6 状态转移与环境边界说明

当前主线训练按时间顺序遍历所有 worker 到达事件。也就是说，`step()` 后的下一条样本可能属于另一个 worker。因此当前环境更接近离线推荐或 contextual bandit 形式的强化学习包装，而不是完全在线 MDP。

环境侧只负责按数据顺序返回 `next_state`，不负责设置折扣因子或训练参数。关于 `gamma` 等训练超参数，由模型训练模块决定。环境另外提供 `load_worker_episodes()`，如果后续要研究同一个 worker 的长期行为，模型组可以读取 `worker_episodes.pkl`，将一个 worker 的完整序列作为一个 episode。

### 3.7 环境 smoke test

使用随机策略验证环境可运行。

训练集 worker 目标：

```text
split: train
reward_type: worker
state_shape: (20, 13)
action_dim: 20
steps: 1000
hit_rate: 0.099
avg_reward: 0.099
```

训练集 requester_urgency 目标：

```text
split: train
reward_type: requester_urgency
state_shape: (20, 13)
action_dim: 20
steps: 1000
hit_rate: 0.099
avg_reward: 0.0043
```

测试集 worker 目标：

```text
split: test
reward_type: worker
state_shape: (20, 13)
action_dim: 20
steps: 1000
hit_rate: 0.043
avg_reward: 0.043
```

由于 worker 奖励是 0/1 命中奖励，因此 `avg_reward` 与 `hit_rate` 相同。测试集随机策略 hit_rate 约为 0.043，可作为后续 DQN 模型的随机基线。
