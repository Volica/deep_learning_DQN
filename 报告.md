# 众包任务推荐强化学习实验报告

## 1. 项目目标与总体思路

本项目研究如何将 DQN 系列强化学习方法应用到众包任务推荐中。当 worker 进入平台时，系统从候选 project 中推荐一个任务。推荐策略需要同时考虑两类利益：

- 参与者利益：worker 能看到更相关、更感兴趣、更可能完成的任务。
- 请求者利益：任务能获得更多、更高质量的回答，尤其是临近截止的任务能被及时推荐。

因此，我们将问题建模为强化学习决策过程：

```text
state = 当前 worker 与候选 project 的匹配特征
action = 推荐候选列表中的一个 project
reward = 根据参与者目标或请求者目标计算反馈
next_state = 推进到下一条按时间排序的 worker 到达事件
```

整体流程分为五个部分：数据清洗与特征工程、强化学习环境设计、DQN 模型实现与训练、参数调优与多变体对比、baseline 与报告分析。

## 2. 成员 A：数据清洗与特征工程

成员 A 负责将原始众包数据整理成可供强化学习环境直接读取的 event 数据。

原始数据包括：

- `data/project/`：project 基本信息，包括类别、行业、开始时间、deadline、entry 数等。
- `data/entry/`：worker 参与 project 的历史记录。
- `data/project_list.csv`：project 列表。
- `data/worker_quality.csv`：worker 质量属性。

预处理脚本为 `data_processed/data_process.py`，输出目录为 `data_processed/`。主要输出文件如下：

| 文件                    | 内容                           |
| ----------------------- | ------------------------------ |
| `enhanced_train.pkl`  | 训练集 event，按时间前 70%     |
| `enhanced_val.pkl`    | 验证集 event，按时间 70% - 85% |
| `enhanced_test.pkl`   | 测试集 event，按时间后 15%     |
| `worker_episodes.pkl` | 按 worker_id 分组的完整序列    |
| `worker_features.pkl` | worker 静态特征                |
| `norm.json`           | 训练集归一化参数               |
| `category_maps.json`  | 类别、子类别、行业映射表       |
| `info.json`           | 数据集规模和特征维度信息       |

数据集规模如下：

| 数据集 | 样本数 |
| ------ | -----: |
| train  | 134475 |
| val    |  28816 |
| test   |  28817 |

单条样本结构如下：

```python
{
    "worker_id": int,
    "timestamp": datetime,
    "candidate_projects": List[int],
    "candidate_features": List[Dict],
    "positive_project": int,
    "positive_index": int,
    "worker_history_at_t": List[int],
}
```

其中 `positive_project` 表示 worker 在历史数据中实际选择的项目，`positive_index` 表示该项目在候选列表中的下标，是训练奖励和 hit_rate 评估的重要依据。

成员 A 构造了 13 维候选项目特征：

```text
worker_quality
worker_history_count
worker_active_days
project_category
project_sub_category
project_industry
project_entry_count
project_duration_days
category_match
industry_match
project_popularity
worker_category_count
remaining_days
```

其中 `worker_category_count` 表示当前时刻之前 worker 在该类别上的历史完成次数，体现 worker 的专项能力；`remaining_days` 表示 project 距离 deadline 的剩余天数，用于刻画任务紧迫度和过滤过期任务。

除类别特征、二值匹配特征和 `remaining_days` 外，连续特征使用训练集统计量做 z-score 标准化。计数类长尾特征先做 `log1p`，再进行标准化，避免大数值特征主导模型训练。

## 3. 成员 B：强化学习环境设计

成员 B 负责 `env/` 文件夹，实现众包任务推荐环境 `CrowdRecEnv`，将预处理后的数据封装为 DQN 可以交互的形式。本部分只描述环境中已经实现的接口、状态构造、动作合法性判断和奖励函数；模型网络、训练循环和调参不属于成员 B 的实现范围。

### 3.1 环境读取接口

环境默认读取 `data_processed/` 中的数据，提供以下接口：

```python
load_split("train")
load_split("val")
load_split("test")
load_all_splits()
load_worker_episodes()
```

其中 `load_worker_episodes()` 用于读取按 worker_id 组织的 episode 数据，便于后续扩展为 worker 序列级强化学习。

### 3.2 状态设计

环境 state 是一个字典：

```python
{
    "features": np.ndarray,      # shape = [20, 13]
    "action_mask": np.ndarray,   # shape = [20]
    "worker_id": np.ndarray,
    "timestamp_index": np.ndarray
}
```

`features` 是候选任务特征矩阵。每一行表示一个候选 project，每一列表示固定顺序的特征。当前新版数据中最大候选数为 20，特征维度为 13，因此：

```text
state["features"].shape = [20, 13]
action_dim = 20
```

如果某条样本候选数不足 20，环境会 padding 到固定长度，并通过 `action_mask` 屏蔽 padding 位置。

### 3.3 动作设计

动作定义为候选项目下标：

```text
action ∈ [0, 19]
```

例如 `action = 3` 表示推荐当前候选列表中的第 4 个 project。DQN 输出长度为 20 的 Q 值向量，每个位置对应一个候选任务。

### 3.4 过期任务过滤

根据大作业要求，系统应考虑任务创建时间、到期时间和 worker 到达时间。环境使用 `remaining_days` 过滤已经过期的候选任务：

```text
remaining_days <= 0 时，action_mask 对应位置设为 0
```

如果旧数据没有 `remaining_days`，环境会尝试使用 `deadline` 和当前样本 `timestamp` 判断；如果两者都不存在，则默认该候选任务可推荐。

### 3.5 奖励设计

环境支持四种奖励类型。

参与者目标：

```text
reward = 1, 如果 action == positive_index
reward = 0, 否则
```

该奖励衡量推荐是否命中 worker 历史真实选择的任务。

请求者质量目标：

```text
reward = worker_quality, 如果 action == positive_index
reward = 0, 否则
```

该奖励鼓励将任务推荐给质量更高的 worker。

请求者紧迫度目标：

```text
reward = worker_quality + urgency_weight * urgency
urgency = 1 / (1 + exp(remaining_days))
```

当任务越接近 deadline，`remaining_days` 越小，`urgency` 越大。

混合目标：

```text
reward = alpha * worker_reward + (1 - alpha) * requester_reward
```

该奖励用于同时考虑参与者和请求者收益。

### 3.6 状态转移与环境边界说明

当前主线训练按时间顺序遍历所有 worker 到达事件。也就是说，`step()` 后的下一条样本可能属于另一个 worker。因此当前环境更接近离线推荐或 contextual bandit 形式的强化学习包装，而不是完全在线 MDP。

环境侧只负责按数据顺序返回 `next_state`，不负责设置折扣因子或训练参数。关于 `gamma` 等训练超参数，由模型训练模块决定。环境另外提供 `load_worker_episodes()`，如果后续要研究同一个 worker 的长期行为，模型组可以读取 `worker_episodes.pkl`，将一个 worker 的完整序列作为一个 episode。

### 3.7 环境 smoke test

使用随机策略验证环境可运行。

训练集 worker 目标：

```text
split: train
reward_type: worker
state_shape: (20, 13)
action_dim: 20
steps: 1000
hit_rate: 0.099
avg_reward: 0.099
```

训练集 requester_urgency 目标：

```text
split: train
reward_type: requester_urgency
state_shape: (20, 13)
action_dim: 20
steps: 1000
hit_rate: 0.099
avg_reward: 0.0043
```

测试集 worker 目标：

```text
split: test
reward_type: worker
state_shape: (20, 13)
action_dim: 20
steps: 1000
hit_rate: 0.043
avg_reward: 0.043
```

由于 worker 奖励是 0/1 命中奖励，因此 `avg_reward` 与 `hit_rate` 相同。测试集随机策略 hit_rate 约为 0.043，可作为后续 DQN 模型的随机基线。

## 4. 成员 C：DQN 主体模型实现与 worker 目标训练

成员 C 负责 worker 模型侧的 DQN 主体实现，将成员 B 提供的环境状态接入神经网络，并完成以 worker 利益最大化为目标的训练、评估和模型保存。本部分对应的主要代码包括 `models.py`、`agent.py`、`replay_buffer.py`、`train.py`、`evaluate.py` 和 `metrics.py`。

### 4.1 Q 函数构造

在本任务中，系统每次只向 worker 推荐一个任务，因此 DQN 的动作不是连续控制动作，而是候选任务列表中的下标：

```text
action ∈ [0, 19]
```

环境返回的状态中，最核心的是候选任务特征矩阵：

```text
state["features"].shape = [20, 13]
```

其中 20 表示最多 20 个候选 project，13 表示每个候选 project 的特征维度。第 `i` 行记为 `feature_i`，表示当前 worker 与第 `i` 个候选任务之间的匹配情况。

成员 C 采用候选任务逐个打分的方式构造 Q 函数：

```text
Q(s, a_i) = MLP(feature_i)
```

也就是说，Q 网络不是把第 0 个动作、第 1 个动作当成固定语义的位置来学习，而是让同一个 MLP 共享参数，对每一个候选任务的特征向量分别输出一个 Q 值。这样更符合推荐任务的特点：动作的含义来自候选 project 本身，而不是它在 padding 后列表中的位置。

网络输出为：

```text
q_values = [Q(s, a_0), Q(s, a_1), ..., Q(s, a_19)]
```

最终推荐策略为：

```text
argmax_i Q(s, a_i)
```

为了避免模型选到 padding 位置或已经过期的任务，动作选择时会使用 `action_mask`。当 `action_mask[i] = 0` 时，该位置对应的 Q 值会被置为一个极小值，使模型只能在合法候选任务中选择。

### 4.2 模型结构与核心模块

成员 C 实现并整理了 DQN 训练所需的几个模块。

| 模块 | 作用 |
| ---- | ---- |
| `models.py` | 定义基础 Q 网络、候选任务共享打分网络和 Dueling Q 网络 |
| `agent.py` | 封装 DQN Agent，包括动作选择、经验存储、参数更新和模型保存 |
| `replay_buffer.py` | 实现经验回放池，保存 `(state, action, reward, next_state, done)` |
| `train.py` | 单次训练入口，负责读取数据、创建环境、训练模型和验证评估 |
| `evaluate.py` | 加载 `best_model.pt` 或 `last_model.pt`，在验证集或测试集上评估 |
| `metrics.py` | 将每个 epoch 的 loss、reward、hit rate 等指标写入 CSV/JSON |

训练时，Agent 使用 epsilon-greedy 策略进行探索。训练初期 `epsilon` 较大，模型会更多随机选择合法动作；随着训练步数增加，`epsilon` 逐渐衰减，模型更多根据 Q 值选择任务。

经验回放池用于打乱样本之间的时间相关性。每一步交互后，成员 C 将以下内容写入 replay buffer：

```python
state_features
state_mask
action
reward
next_features
next_mask
done
positive_action
```

其中 `positive_action` 就是 `positive_index`，表示历史数据中 worker 真实选择的任务下标。

### 4.3 DQN 与 Double DQN 更新

基础 DQN 的目标值为：

```text
y = r + gamma * max_a' Q_target(s', a')
```

Double DQN 将“动作选择”和“动作估值”分开，先用 online network 选择下一步最优动作，再用 target network 估计该动作的 Q 值：

```text
a* = argmax_a' Q_online(s', a')
y  = r + gamma * Q_target(s', a*)
```

当前实验中主线任务更接近离线推荐场景或 contextual bandit，因此默认 `gamma = 0.0`。这样模型重点学习当前 worker 到达时推荐哪个任务可以立即获得更高奖励，而不过度依赖离线日志中并不完全可控的长期转移。

损失函数使用 Huber loss：

```text
TD_loss = SmoothL1Loss(Q(s, a), y)
```

为了让模型更稳定地学习历史真实选择的位置，成员 C 还加入了辅助交叉熵损失：

```text
CE_loss = CrossEntropy(masked_q_values, positive_index)
loss = TD_loss + aux_ce_weight * CE_loss
```

其中 `masked_q_values` 会先用 `action_mask` 屏蔽非法动作，再与 `positive_index` 计算交叉熵。`aux_ce_weight = 0` 时相当于纯 DQN；`aux_ce_weight > 0` 时，模型同时利用强化学习奖励和历史选择监督信号。

### 4.4 Worker 目标训练与评估

worker 目标关注参与者是否能看到自己更可能感兴趣并完成的任务，因此奖励定义为：

```text
reward = 1, 如果 action == positive_index
reward = 0, 否则
```

在这个目标下，平均奖励与 hit rate 含义一致，都是衡量模型推荐任务是否命中 worker 历史真实选择。

成员 C 在训练中记录以下指标：

| 指标 | 含义 |
| ---- | ---- |
| `train_avg_reward` | 训练集平均奖励 |
| `train_hit_rate` | 训练集命中率 |
| `eval_avg_reward` | 验证集平均奖励 |
| `eval_hit_rate` | 验证集命中率 |
| `loss` | 每个 epoch 的平均训练损失 |
| `epsilon` | 当前探索率 |
| `eval_invalid_action_rate` | 非法动作比例 |

每次训练会保存：

```text
config.json
metrics.csv
summary.json
best_model.pt
last_model.pt
```

其中 `best_model.pt` 根据验证集平均奖励选择，用于最终测试集评估。这样可以避免只看训练集效果导致过拟合。

## 5. 成员 D：Requester 目标建模、参数调优与实验对比分析

成员 D 负责 requester 模型侧的目标设计、参数调优和实验指标分析。该部分在成员 C 的 DQN 主体训练框架基础上，重点研究如何让推荐结果更符合请求者利益，并通过统一条件下的参数实验分析模型表现。主要代码包括 `requester_dqn.py` 和参数实验相关训练入口。

### 5.1 Requester 目标与奖励函数

requester 的利益并不只是“worker 是否点击或完成任务”，还包括任务能否被高质量 worker 完成，以及临近截止的任务能否及时得到回答。因此成员 D 将 requester 奖励设计为命中真实选择后，再结合 worker 质量和任务紧迫度：

```text
reward = worker_quality + urgency_weight * urgency
urgency = 1 / (1 + exp(remaining_days))
```

其中 `remaining_days` 表示 worker 到达时刻距离任务 deadline 的剩余天数。剩余时间越短，`remaining_days` 越小，`urgency` 越大，模型越倾向于优先推荐紧急任务。

在扩展实验中，还可以加入项目热度项：

```text
reward = quality_weight * worker_quality
       + urgency_weight * urgency
       + popularity_weight * project_popularity
```

其中 `project_popularity = entry_count / duration_days`，用于描述项目在生命周期内吸引 worker 的程度。该项默认可以设为 0，避免热门项目过度挤占长尾任务。

### 5.2 数据分析与特征构造思路

成员 D 在 requester 目标中重点分析了三类信息。

第一类是 worker 侧特征：

```text
worker_quality
worker_history_count
worker_active_days
worker_category_count
```

`worker_quality` 直接反映 worker 的历史质量；`worker_history_count` 和 `worker_active_days` 表示 worker 的经验和活跃程度；`worker_category_count` 表示当前时刻之前 worker 在候选 project 所属类别上的完成次数，用来模拟 worker 对某类任务的熟悉程度。

第二类是 project 侧特征：

```text
project_category
project_sub_category
project_industry
project_entry_count
project_duration_days
project_popularity
remaining_days
```

其中 `remaining_days` 是 requester 目标中最关键的动态特征。它不是 project 的静态属性，而是由任务 deadline 和 worker 到达时间共同决定。

第三类是 worker-project 匹配特征：

```text
category_match
industry_match
```

这两个特征用于刻画 worker 历史偏好与候选 project 是否匹配。如果 worker 过去经常完成某一类别或行业的任务，则该类任务更可能被 worker 接受，也更可能为 requester 带来有效回答。

### 5.3 时间排序与动态模拟

众包平台中的任务和 worker 都是动态变化的，因此成员 D 在实验分析中特别强调时间顺序。数据构造和环境交互遵循以下逻辑：

```text
1. 按 worker 的 entry_created_at 对历史交互排序
2. 当 worker 在时刻 t 到达平台时，只考虑 start_date <= t <= deadline 的 project
3. 对每个候选 project 计算 remaining_days = deadline - t
4. 如果 remaining_days <= 0，则通过 action_mask 将该任务设为不可推荐
5. worker 做出选择后，进入下一条按时间排序的 worker 到达事件
```

这样可以同时模拟任务创建时间、任务到期时间和 worker 到达时间。模型看到的不是全局静态任务池，而是在某个时刻真实可推荐的候选任务集合。

worker 的兴趣也通过时间动态更新。对于每个 worker，只统计当前时刻之前完成过的任务，构造 `worker_history_at_t` 和 `worker_category_count`。因此模型不会提前看到未来行为，避免了数据泄露。

### 5.4 参数调优与指标记录

成员 D 负责在统一训练条件下完成参数调优，不负责训练曲线、验证曲线和测试集柱状图的生成。可视化部分由成员 E 根据 `metrics.csv` 进一步处理，成员 D 只保证每组实验的参数设置、指标记录和结果对比口径一致。

调参时保持数据划分、epoch、batch size、epsilon 衰减步数、target network 更新间隔等条件一致，重点比较学习率：

| 学习率 | 观察重点 |
| ------ | -------- |
| `0.01` | loss 是否震荡，验证集 hit rate 是否不稳定 |
| `0.001` | 训练速度和验证集效果是否平衡 |
| `0.0001` | loss 是否更平滑，是否存在收敛较慢的问题 |

除学习率外，成员 D 也记录了以下可调参数对结果的影响：

| 参数 | 作用 |
| ---- | ---- |
| `batch_size` | 控制梯度估计稳定性 |
| `epsilon_decay_steps` | 控制探索持续时间 |
| `aux_ce_weight` | 控制历史选择监督信号的强弱 |
| `hidden_dims` | 控制 Q 网络容量 |

所有实验统一写入 `metrics.csv`，并用 `eval_hit_rate`、`eval_avg_reward`、`loss` 和 `eval_invalid_action_rate` 判断模型效果。成员 D 在调参结论中只比较数值指标，不负责图片生成与版面展示。

| `worker_test.eval_hit_rate` | 参与者目标下测试集命中率 |
| `requester_test.eval_hit_rate` | 请求者目标下测试集命中率 |
| `worker_val_best_hit` | worker 目标验证集最佳命中率 |
| `requester_val_best_hit` | requester 目标验证集最佳命中率 |
| `loss` | 训练稳定性 |

基础 DQN 直接用 target network 计算最大下一步 Q 值；Double DQN 分离动作选择和动作估值，减少 Q 值过估计；Dueling DQN 将状态价值和动作优势拆开建模，适合分析不同候选任务在同一状态下的相对优势。

最终，成员 E 通过训练曲线、测试集柱状图和多变体汇总表，分析 requester 目标与 worker 目标之间的差异：worker 目标更关注是否命中个人兴趣，requester 目标则进一步强调高质量 worker、任务紧迫度和任务完成价值。
