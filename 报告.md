# 众包任务推荐强化学习实验报告

## 1. 项目目标与总体思路

本项目研究如何将 DQN 系列强化学习方法应用到众包任务推荐中。当 worker 进入平台时，系统从候选 project 中推荐一个任务。推荐策略需要同时考虑两类利益：

- 参与者利益：worker 能看到更相关、更感兴趣、更可能完成的任务。
- 请求者利益：任务能获得更多、更高质量的回答，尤其是临近截止的任务能被及时推荐。

因此，我们将问题建模为强化学习决策过程：

```text
state = 当前 worker 与候选 project 的匹配特征
action = 推荐候选列表中的一个 project
reward = 根据参与者目标或请求者目标计算反馈
next_state = 推进到下一条按时间排序的 worker 到达事件
```

整体流程分为五个部分：数据清洗与特征工程、强化学习环境设计、DQN 模型实现与训练、参数调优与多变体对比、baseline 与报告分析。

## 2. 成员 A：数据清洗与特征工程

成员 A 负责将原始众包数据预处理成可供强化学习环境直接读取的 event 数据。原始数据包括 `data/project/` 中的 project 基本信息、`data/entry/` 中的 worker 参与记录、`data/project_list.csv` 和 `data/worker_quality.csv`。根据后续模型目标，数据处理分为 worker 视角和 requester 视角两部分。

### 2.1 Worker 视角数据处理

Worker 视角预处理脚本为 `data_processed/data_process.py`。该脚本以 worker 到达平台为决策点，构造“给当前 worker 推荐 candidate project”的样本，并按时间顺序切分训练集、验证集和测试集。

主要输出文件包括：

| 文件                    | 内容                                      |
| ----------------------- | ----------------------------------------- |
| `enhanced_train.pkl`  | worker 视角训练集 event，按时间前 70%     |
| `enhanced_val.pkl`    | worker 视角验证集 event，按时间 70% - 85% |
| `enhanced_test.pkl`   | worker 视角测试集 event，按时间后 15%     |
| `worker_episodes.pkl` | 按 worker_id 分组的完整序列               |
| `worker_features.pkl` | worker 静态特征                           |
| `norm.json`           | 训练集归一化参数                          |
| `category_maps.json`  | 类别、子类别、行业映射表                  |
| `info.json`           | worker 视角数据集规模和特征维度信息       |

Worker 视角数据集规模如下：

| 数据集 | 样本数 |
| ------ | -----: |
| train  | 134475 |
| val    |  28816 |
| test   |  28817 |

单条 worker 样本结构如下：

```python
{
    "worker_id": int,
    "timestamp": datetime,
    "candidate_projects": List[int],
    "candidate_features": List[Dict],
    "positive_project": int,
    "positive_index": int,
    "worker_history_at_t": List[int],
}
```

其中 `positive_project` 表示 worker 在历史数据中实际选择的项目，`positive_index` 表示该项目在候选列表中的下标，是训练奖励和 hit_rate 评估的重要依据。

为每个 candidate project 构造 13 维特征：

```text
worker_quality
worker_history_count
worker_active_days
project_category
project_sub_category
project_industry
project_entry_count
project_duration_days
category_match
industry_match
project_popularity
worker_category_count
remaining_days
```

其中 `worker_category_count` 表示当前时刻之前 worker 在该类别上的历史完成次数，体现 worker 的专项能力；`remaining_days` 表示 project 距离 deadline 的剩余天数，用于刻画任务紧迫度和过滤过期任务。类别特征和行业特征使用映射表编码，计数类长尾特征先做 `log1p`，再使用训练集统计量进行 z-score 标准化。

### 2.2 Requester 视角数据处理

Requester 视角预处理脚本为 `data_processed/data_process_request.py`。该脚本以 project 为中心构造样本：每个 event 表示某个 project 在某一时刻收到一条新 entry，模型需要从当前候选 worker 池中推荐合适的 worker。

主要输出文件包括：

| 文件                     | 内容                                 |
| ------------------------ | ------------------------------------ |
| `requester_train.pkl`  | requester 视角训练集 event           |
| `requester_val.pkl`    | requester 视角验证集 event           |
| `requester_test.pkl`   | requester 视角测试集 event           |
| `project_episodes.pkl` | 按 project_id 分组的完整序列         |
| `worker_features.pkl`  | 与 worker 视角共享的 worker 静态特征 |

Requester 单条样本包含：

```python
{
    "project_id": int,
    "timestamp": datetime,
    "candidate_workers": List[int],
    "candidate_features": List[Dict],
    "positive_worker": int,
    "positive_index": int,
    "project_history_at_t": List[int],
}
```

其中 `candidate_workers` 由负样本和一个正样本组成，`positive_worker` 表示当前时刻实际提交 entry 的 worker，`positive_index` 表示该 worker 在候选列表中的下标。`project_history_at_t` 记录当前时刻之前该 project 已经收到的 worker 序列，用于刻画 project 的动态完成状态。

Requester 视角使用 16 维候选 worker 特征：

```text
project_category
project_sub_category
project_industry
project_duration_days
project_remaining_days
project_current_entry_count
project_hist_entry_count
project_fill_rate
worker_quality
worker_history_count
worker_active_days
worker_category_match
worker_industry_match
worker_category_count
worker_already_submitted
worker_recent_activity
```

其中 project 侧特征描述当前任务类别、剩余时间、历史与当前参与情况；worker 侧特征描述候选 worker 的质量、经验、活跃度以及与当前 project 的类别和行业匹配关系。`project_current_entry_count`、`project_fill_rate`、`worker_already_submitted` 和 `worker_recent_activity` 都是按时间顺序计算的动态特征，用于避免数据泄露并反映当前决策时刻的真实状态。Requester 数据也按时间顺序划分 train、val、test，并输出 `project_episodes.pkl` 以支持后续 project 序列级训练。

### 2.3 数据分析、特征构造与时间线模拟

为了让强化学习环境中的状态尽可能接近真实众包平台的推荐场景，预处理阶段首先分析原始数据中可用于描述“任务、worker、交互时间”的字段，再按时间顺序构造离线 RL event。原始数据主要包含三类信息：`data/project/` 中的 project 文件提供任务类别、子类别、行业、创建时间 `start_date`、到期时间 `deadline` 和历史 entry 数；`data/entry/` 中的 entry 文件提供 worker 的提交时间 `entry_created_at` 和实际参与记录；`worker_quality.csv` 提供 worker 的历史质量分。由于原始日志直接记录的是提交时间，本文将 `entry_created_at` 作为离线模拟中 worker 到达并完成一次交互的观测时刻。

#### 2.3.1 原始数据分析与特征来源

根据上述数据来源，特征被划分为四类。第一类是任务静态特征，包括 `project_category`、`project_sub_category`、`project_industry`、`project_duration_days` 和历史 `project_entry_count`，用于描述任务本身的类型、领域和历史热度。第二类是 worker 静态与历史特征，包括 `worker_quality`、`worker_history_count` 和 `worker_active_days`，用于描述 worker 的历史质量、经验和活跃程度。第三类是 worker 与任务之间的匹配特征，包括 `category_match`、`industry_match`、`worker_category_count`、`worker_category_match` 和 `worker_industry_match`，用于刻画 worker 是否更可能对某类任务感兴趣或更擅长完成该类任务。第四类是动态时间与完成状态特征，包括 `remaining_days`、`project_remaining_days`、`project_current_entry_count`、`project_fill_rate`、`worker_already_submitted` 和 `worker_recent_activity`，用于描述当前时刻任务是否紧急、任务完成进度如何，以及候选 worker 是否已经参与过同一任务。

这些特征共同服务于两个目标：在 worker 视角下，模型需要判断当前 worker 更可能选择并完成哪个 project；在 requester 视角下，模型需要判断当前 project 更适合推荐给哪个 worker，从而获得更高质量、更及时且更不重复的提交。

#### 2.3.2 基于时间排序的事件构建

事件构建时，首先解析每个 project 的 `start_date` 和 `deadline`，得到任务的有效时间窗口；然后解析所有 entry 的 `entry_created_at`，将所有 worker-project 交互记录按时间升序排序。排序后的每一条交互记录都被视为一个决策事件，表示在时刻 `t`，某个 worker 与某个 project 发生了一次真实提交行为。

在 worker 视角，处理时刻 `t` 的事件时，只将满足 `start_date <= t <= deadline` 的 project 放入候选集，保证候选任务在当前时刻已经发布且尚未截止。当前日志中实际被该 worker 提交的 project 被设为 `positive_project`，其在候选列表中的位置为 `positive_index`。环境中的动作就是从这些候选 project 中选择一个进行推荐。如果模型选择的位置等于 `positive_index`，则说明推荐命中了历史中 worker 实际完成的任务。

在 requester 视角，处理时刻 `t` 的事件时，将当前 project 实际收到提交的 worker 作为 `positive_worker`，再从 worker 池中采样负样本，组成 `candidate_workers`。这样每个 requester event 都表示“当前 project 在该时刻需要从候选 worker 中选择一个推荐对象”的决策问题。

#### 2.3.3 Worker 兴趣与 Requester 完成情况的模拟

为了模拟 worker 的兴趣和完成情况，预处理脚本按时间维护每个 worker 的历史任务序列 `worker_history_at_t`。在构造时刻 `t` 的状态时，只使用 `t` 之前的历史记录计算 `worker_history_count`、`worker_category_count`、类别匹配和行业匹配等特征；当前真实提交的 project 会在该事件结束后再加入 worker 历史。这样可以避免使用未来信息，同时让后续状态能够反映 worker 兴趣随历史完成任务逐步变化的过程。

为了模拟 requester 关心的任务完成状态，预处理脚本按时间维护每个 project 已收到的 worker 序列 `project_history_at_t` 和当前 entry 数 `project_current_entry_count`。每处理一条 entry 后，当前 worker 会被加入该 project 的历史参与列表，entry 数和 `project_fill_rate` 也随之更新。`project_remaining_days` 则由 `deadline - t` 得到，用于刻画任务距离截止时间的紧迫程度。通过这些动态变量，模型可以区分尚未充分完成、临近截止、已经有候选 worker 提交过等不同状态，从而在 requester 目标下更重视高质量 worker、任务紧急度和参与者多样性。

## 3. 成员 B：强化学习环境与状态构造

成员 B 负责 `env/` 目录中的环境封装，核心文件为 `crowd_env.py`。本部分的工作是把预处理后的样本转换为 DQN 可以直接交互的形式，包括数据读取接口、状态矩阵、动作空间、合法动作过滤和状态转移。worker 奖励使用环境默认的命中奖励；requester 奖励由成员 D 在 requester 模型侧设计和说明。

### 3.1 环境接口

当前环境保留两套视角。

| 环境                  | 数据文件                         | 决策含义                                                   |
| --------------------- | -------------------------------- | ---------------------------------------------------------- |
| `CrowdRecEnv`       | `enhanced_train/val/test.pkl`  | 一个 worker 到达时，从候选 project 中推荐一个任务          |
| `RequesterCrowdEnv` | `requester_train/val/test.pkl` | 一个 project 需要参与者时，从候选 worker 中推荐一个 worker |

对应的数据读取函数包括：

```python
load_split("train")
load_split("val")
load_split("test")
load_requester_split("train")
load_requester_split("val")
load_requester_split("test")
load_worker_episodes()
load_project_episodes()
```

其中 `load_worker_episodes()` 和 `load_project_episodes()` 用于按 worker 或 project 聚合的序列数据，便于后续扩展为更完整的 episode 训练。

### 3.2 Worker 视角

`CrowdRecEnv` 的状态是一个字典：

```python
{
    "features": np.ndarray,
    "action_mask": np.ndarray,
    "worker_id": np.ndarray,
    "timestamp_index": np.ndarray
}
```

`features` 是候选 project 的特征矩阵。每一行对应一个候选任务，每一列按固定顺序存放特征。当前 worker 视角特征为 13 维：

```text
worker_quality
worker_history_count
worker_active_days
project_category
project_sub_category
project_industry
project_entry_count
project_duration_days
category_match
industry_match
project_popularity
worker_category_count
remaining_days
```

最大候选数为 20，因此主线 worker 环境的状态形状为：

```text
state["features"].shape = [20, 13]
state["action_mask"].shape = [20]
action_dim = 20
```

如果某条样本候选数不足 20，环境会用 0 padding 到固定长度，并用 `action_mask` 屏蔽 padding 位置。`remaining_days` 在环境中会经过 sigmoid 缩放，避免原始天数尺度过大影响模型输入。

Worker 视角的动作是候选 project 的下标：

```text
action ∈ [0, 19]
```

例如 `action = i` 表示向当前 worker 推荐第 `i` 个 candidate project。环境用 `remaining_days >= 0` 判断任务是否仍可推荐；如果某个位置是 padding，或者任务已经超过可推荐时间，则 `action_mask` 对应位置为 0。

`step(action)` 会检查动作是否合法，计算当前样本的 worker 默认奖励，写入 `info`，然后把环境索引推进到下一条按时间排序的 worker 到达事件。worker 默认奖励为：

```text
reward = 1, 如果 action == positive_index
reward = 0, 否则
```

其中 `positive_index` 是历史数据中 worker 实际选择的任务在候选列表中的位置。因此在 worker 目标下，平均奖励与 hit rate 含义一致。

### 3.3 Requester 视角

环境中还包括Requester视角下的 `RequesterCrowdEnv`。它的候选对象不再是 project，而是 `candidate_workers`。每条样本表示某个 project 在当前时刻需要推荐 worker，模型从候选 worker 列表中选择一个。

Requester 视角使用 16 维特征：

```text
project_category
project_sub_category
project_industry
project_duration_days
project_remaining_days
project_current_entry_count
project_hist_entry_count
project_fill_rate
worker_quality
worker_history_count
worker_active_days
worker_category_match
worker_industry_match
worker_category_count
worker_already_submitted
worker_recent_activity
```

这些特征同时包含 project 当前状态、候选 worker 的历史质量、worker 与 project 的匹配程度，以及 `worker_already_submitted`、`worker_recent_activity` 这类动态行为特征。环境返回的 state 同样包含 `features` 和 `action_mask`，并额外返回 `project_id` 用于标识当前决策点。

Requester 视角的动作也是候选列表下标：

```text
action ∈ [0, 19]
```

但这里 `action = i` 表示向当前 project 推荐第 `i` 个 candidate worker。环境使用 `project_remaining_days >= 0` 构造 `action_mask`，并屏蔽 padding 位置。`step(action)` 同样会推进到下一条 requester 决策样本，并在 `info` 中返回 `project_id`、`chosen_worker`、`positive_worker`、`positive_index` 等字段。Requester 的具体奖励由成员 D 设计，因此本节只说明环境如何承载动作和状态转移。

整体来看，两个环境的转移都更接近离线推荐场景中的 contextual bandit 包装：每个样本是一个独立推荐决策点，`step()` 后进入下一条历史事件。环境负责按数据顺序推进，不负责设置 `gamma`、batch size 或 target network 更新等训练超参数。

## 4. 成员 C：DQN 主体模型实现与 worker 目标训练

成员 C 负责 DQN 主体模型实现，将成员 B 提供的环境状态接入神经网络，并完成以 worker 利益最大化为目标的训练、评估和模型保存。本部分对应的主要代码包括 work_model/`models.py`、`agent.py`、`replay_buffer.py`、`train.py`、`evaluate.py` 和 `metrics.py`。当前代码符合 DQN 的核心结构：使用神经网络近似 Q 函数，使用经验回放训练，使用实时更新的在线 Q 网络和阶段性同步的目标 Q 网络。

### 4.1 Q 函数构造

在本任务中，系统每次只推荐一个候选对象，因此动作不是连续控制动作，而是候选列表中的下标：

```text
action ∈ [0, 19]
```

worker 环境返回的核心状态是候选任务特征矩阵：

```text
state["features"].shape = [20, 13]
```

其中 20 表示最多 20 个候选 project，13 表示每个候选 project 的特征维度。第 `i` 行记为 `feature_i`，表示当前 worker 与第 `i` 个候选任务之间的匹配情况。

默认的 `QNetwork` 采用候选对象逐个打分的方式构造 Q 函数：

```text
Q(s, a_i) = MLP(feature_i)
```

也就是说，Q 网络不是把第 0 个动作、第 1 个动作当成固定语义的位置来学习，而是让同一个 MLP 共享参数，对每一个候选任务的特征向量分别输出一个 Q 值。这样更符合推荐任务的特点：动作的含义来自候选 project 本身，而不是它在 padding 后列表中的位置。

网络输出为：

```text
q_values = [Q(s, a_0), Q(s, a_1), ..., Q(s, a_19)]
```

最终推荐策略为：

```text
argmax_i Q(s, a_i)
```

为了避免模型选到 padding 位置或已经过期的任务，动作选择时会使用 `action_mask`。当 `action_mask[i] = 0` 时，该位置对应的 Q 值会被置为一个极小值，使模型只能在合法候选任务中选择。

### 4.2 模型结构与核心模块

成员 C 实现并整理了 DQN 训练所需的几个模块。

| 模块                 | 作用                                                                |
| -------------------- | ------------------------------------------------------------------- |
| `models.py`        | 定义候选共享打分 Q 网络、旧版扁平 Q 网络和 Dueling Q 网络           |
| `agent.py`         | 封装 DQN Agent，包括动作选择、经验存储、在线网络更新和目标网络同步  |
| `replay_buffer.py` | 实现经验回放池，保存 `(state, action, reward, next_state, done)`  |
| `train.py`         | worker 单目标训练入口，负责读取数据、创建环境、训练模型和验证评估   |
| `evaluate.py`      | 加载 `best_model.pt` 或 `last_model.pt`，在验证集或测试集上评估 |
| `metrics.py`       | 将每个 epoch 的 loss、reward、hit rate 等指标写入 CSV/JSON          |

训练时，Agent 使用 epsilon-greedy 策略进行探索。训练初期 `epsilon` 较大，模型会更多随机选择合法动作；随着训练步数增加，`epsilon` 逐渐衰减，模型更多根据在线 Q 网络输出的 Q 值选择任务。`DQNAgent.act()` 中会先取出 `action_mask` 允许的位置，再在探索动作或最大 Q 值动作之间选择，保证训练和评估阶段都不会主动选择非法候选。

经验回放池用于打乱样本之间的时间相关性。每一步交互后，成员 C 将以下内容写入 replay buffer：

```python
state_features
state_mask
action
reward
next_features
next_mask
done
positive_action
```

其中 `positive_action` 就是 `positive_index`，表示历史数据中 worker 真实选择的任务下标，用于辅助交叉熵损失。

### 4.3 在线 Q 网络与目标 Q 网络更新

当前 `DQNAgent` 同时维护两套网络：

```text
online_net：实时更新的 Q 网络，用于当前动作选择和梯度学习
target_net：阶段性更新的目标网络，用于计算 TD target
```

初始化时，`target_net` 会复制 `online_net` 的参数。之后每次 `learn()` 从 replay buffer 中采样一个 mini-batch，使用 `online_net(states)` 计算当前动作的 Q 值，并通过反向传播实时更新 `online_net`。代码中的更新流程为：

```text
q_values = online_net(states)
q_selected = q_values[action]
loss.backward()
optimizer.step()
```

目标网络不会每一步都更新，而是每隔 `target_update_interval` 个训练步执行一次：

```text
target_net.load_state_dict(online_net.state_dict())
```

默认 `target_update_interval = 500`。这种“在线网络实时学习、目标网络阶段性同步”的结构正是 DQN 稳定训练的关键，可以避免训练目标随着在线网络每一步更新而剧烈震荡。

基础 DQN 的目标值为：

```text
y = r + gamma * max_a' Q_target(s', a')
```

Double DQN 将“动作选择”和“动作估值”分开，先用 online network 选择下一步最优动作，再用 target network 估计该动作的 Q 值：

```text
a* = argmax_a' Q_online(s', a')
y  = r + gamma * Q_target(s', a*)
```

当前 `work_model/train.py` 中 worker 主线训练默认 `gamma = 0.0`，因为离线推荐样本更接近 contextual bandit，重点是学习当前 worker 到达时推荐哪个任务可以得到即时奖励。`requester_model/visualize_train.py` 中 `gamma` 作为可调参数传入，默认值为 0.99，用于对比实验中保留更完整的 DQN 形式。

损失函数使用 Huber loss：

```text
TD_loss = SmoothL1Loss(Q(s, a), y)
```

为了让模型更稳定地学习历史真实选择的位置，成员 C 还加入了辅助交叉熵损失：

```text
CE_loss = CrossEntropy(masked_q_values, positive_index)
loss = TD_loss + aux_ce_weight * CE_loss
```

其中 `masked_q_values` 会先用 `action_mask` 屏蔽非法动作，再与 `positive_index` 计算交叉熵。`aux_ce_weight = 0` 时相当于纯 DQN；`aux_ce_weight > 0` 时，模型同时利用强化学习奖励和历史选择监督信号。

### 4.4 DQN 变体说明

当前代码支持三类模型变体：

| 变体            | 当前实现                                                              |
| --------------- | --------------------------------------------------------------------- |
| `dqn`         | 使用 `target_net` 计算下一状态最大 Q 值                             |
| `double_dqn`  | 使用 `online_net` 选下一步动作，使用 `target_net` 估值            |
| `dueling_dqn` | 使用 Dueling 结构，将状态价值 `V(s)` 和动作优势 `A(s,a)` 分开建模 |

需要注意的是，默认的 `dqn` 和 `double_dqn` 使用候选共享打分网络 `QNetwork`；`dueling_dqn` 当前使用 `DuelingQNetwork`，会先把整个候选矩阵展开，再输出每个动作位置的 Q 值。因此 Dueling DQN 更适合作为模型变体对比，报告实验结果时应说明它与候选共享打分网络在输入结构上存在差异。

### 4.5 Worker 目标训练与评估

worker 目标关注参与者是否能看到更相关、更感兴趣、更可能完成的任务，从而提高参与者完成任务和获得报酬的机会。因此奖励定义为：

```text
reward = 1, 如果 action == positive_index
reward = 0, 否则
```

在这个目标下，平均奖励与 hit rate 含义一致，都是衡量模型推荐任务是否命中 worker 历史真实选择。由于 worker 视角特征中包含 `category_match`、`industry_match`、`worker_category_count`、`remaining_days` 等信息，模型不只是记忆候选位置，而是利用 worker 偏好、历史经验、任务类别和截止时间来学习推荐策略。

成员 C 在训练中记录以下指标：

| 指标                         | 含义                      |
| ---------------------------- | ------------------------- |
| `train_avg_reward`         | 训练集平均奖励            |
| `train_hit_rate`           | 训练集命中率              |
| `eval_avg_reward`          | 验证集平均奖励            |
| `eval_hit_rate`            | 验证集命中率              |
| `loss`                     | 每个 epoch 的平均训练损失 |
| `epsilon`                  | 当前探索率                |
| `eval_invalid_action_rate` | 非法动作比例              |

每次训练会保存：

```text
config.json
metrics.csv
summary.json
best_model.pt
last_model.pt
```

其中 `best_model.pt` 根据验证集平均奖励选择，用于最终测试集评估。这样可以避免只看训练集效果导致过拟合。

## 5. 成员 D：Requester 目标建模、奖励设计与训练实现

成员 D 负责 requester 模型侧的目标设计和训练实现，使用 `RequesterCrowdEnv` 和 `requester_train/val/test.pkl` 数据，问题定义为请求者视角建模“给当前 project 推荐合适 worker”。主要实现位于 `requester_model/visualize_train.py`，并复用 `work_model.agent` 中的 DQN、Double DQN 和 Dueling DQN 训练框架。

### 5.1 Requester 决策目标

Requester 目标关注的是 project 能否获得更多、更高质量、更加及时且更多样化的 worker 参与。因此 requester 环境中的一个决策点表示：

```text
当前 project 在时刻 t 需要推荐 worker
模型从 candidate_workers 中选择一个 worker
```

与 worker 目标不同，requester 目标不再把“是否命中历史选择”作为唯一优化目标。`hit_rate` 仍然保留为参考指标，但主要优化方向是推荐 worker 的质量、紧急任务下的响应质量，以及避免总是推荐已经参与过同一 project 的 worker。

### 5.2 Requester 奖励函数

在 `RequesterCrowdEnv` 中设计奖励为：

```text
urgency = 1 / (1 + exp(project_remaining_days))

effective_quality_weight
    = quality_weight * (1 + urgency_weight * urgency)

reward
    = effective_quality_weight * worker_quality
      + diversity_weight * (1 - worker_already_submitted)
```

默认权重为：

```text
quality_weight = 1.0
urgency_weight = 0.5
diversity_weight = 0.3
```

该奖励由三部分组成。

第一部分是 worker 质量项。`worker_quality` 是主项，表示候选 worker 的历史质量。质量越高，推荐给 requester 的价值越大。

第二部分是紧急度加成。`project_remaining_days` 越小，`urgency` 越大，`effective_quality_weight` 越高。这样当 project 接近 deadline 时，模型会更重视高质量 worker，而不是只追求普通命中。

第三部分是多样性项。`worker_already_submitted` 表示该 worker 是否已经向当前 project 提交过作品。如果该值为 0，说明推荐的是新的 worker，奖励中会增加 `diversity_weight`。这样可以减少重复推荐同一批 worker 的倾向，让 requester 获得更多不同参与者。

### 5.3 Requester 训练与评估

在 `visualize_train.py` 中实现了 requester 目标训练入口 `train_requester_objective()`。训练代码直接使用成员 A 构造的 requester 视角候选 worker 特征，以及成员 B 在 `RequesterCrowdEnv` 中封装好的状态、动作和 `action_mask`。整体流程如下：

```text
1. 使用 load_requester_split() 读取 requester_train/val/test.pkl
2. 使用 RequesterCrowdEnv 构造 requester 视角环境
3. 使用 DQNAgent 根据 state["features"] 输出每个候选 worker 的 Q 值
4. 按环境提供的 action_mask 屏蔽非法位置
5. 将 state、action、reward、next_state 写入 replay buffer
6. 从 replay buffer 采样 mini-batch，实时更新 online_net
7. 每隔 target_update_interval 步同步一次 target_net
8. 每个 epoch 后在 requester 验证集上评估并保存 best_model.pt
```

网络结构与 worker 目标共用同一套 DQN Agent。对于默认的 `dqn` 和 `double_dqn`，仍采用候选对象共享打分方式：

```text
Q(s, a_i) = MLP(feature_i)
```

区别在于 worker 目标下 `feature_i` 表示候选 project，requester 目标下 `feature_i` 表示候选 worker。这样同一套 DQN 框架可以同时服务两个方向的推荐任务。若选择 `dueling_dqn`，则使用 Dueling 网络对整个候选矩阵建模，再输出每个候选 worker 位置的 Q 值。

Requester 训练过程中除记录通用指标外，还增加了更符合 requester 目标的指标：

| 指标                                                         | 含义                                    |
| ------------------------------------------------------------ | --------------------------------------- |
| `train_avg_reward` / `eval_avg_reward`                   | requester 奖励均值                      |
| `train_hit_rate` / `eval_hit_rate`                       | 是否命中历史正样本，作为参考指标        |
| `train_avg_worker_quality` / `eval_avg_worker_quality`   | 推荐 worker 的平均质量                  |
| `train_diversity_rate` / `eval_diversity_rate`           | 推荐未参与过当前 project 的 worker 比例 |
| `train_urgent_avg_quality` / `eval_urgent_avg_quality`   | 紧急 project 上推荐 worker 的平均质量   |
| `train_invalid_action_rate` / `eval_invalid_action_rate` | 非法动作比例                            |
| `loss`                                                     | 每轮平均训练损失                        |

参数实验保持数据划分、epoch、batch size、epsilon 衰减步数和 target network 更新间隔一致，重点比较学习率、DQN 变体和辅助交叉熵权重对 requester 奖励、worker 质量和多样性指标的影响。训练输出统一保存 `config.json`、`metrics.csv`、`best_model.pt` 和 `last_model.pt`，便于成员 E 进一步生成训练曲线和对比图。

### 5.4 双目标需求与动态性的处理

从需求角度看，当前代码并不是只优化单一点击或命中目标，而是从两个视角分别建模。

worker 目标对应参与者需求：模型通过历史选择正样本、类别/行业匹配、worker 历史完成次数和任务剩余时间等特征，学习把更相关、更可能被完成的任务推荐给 worker，使参与者更容易找到感兴趣的任务并获得报酬。

requester 目标对应请求者需求：模型通过 `worker_quality`、`project_remaining_days`、`project_current_entry_count`、`project_fill_rate`、`worker_already_submitted` 和 `worker_recent_activity` 等动态特征，学习优先推荐高质量、及时且未重复参与的 worker，使任务获得更多、更高质量的回答。

动态性主要体现在三方面。第一，数据预处理按 `entry_created_at` 排序，每个 event 只使用当前时刻之前的 worker 历史和 project 历史。第二，环境中的 `action_mask` 会屏蔽 padding 和已过期候选，使任务到期状态影响当前可选动作。第三，requester 视角会随时间更新 `project_current_entry_count`、`project_fill_rate` 和 `worker_already_submitted`，worker 视角会随时间更新 `worker_history_at_t` 和 `worker_category_count`，从而反映参与者兴趣和任务完成状态的变化。

因此，当前实现能够在实验层面同时覆盖参与者和请求者的需求：`visualize_train.py` 可以在同一次实验中分别训练 worker 目标和 requester 目标，并生成对比指标；`work_model/train.py` 还保留了 `hybrid` 奖励形式，用于在 worker 推荐环境中做加权目标实验。不过需要说明的是，本项目仍然基于历史日志构造离线环境，模型动作不会真正改变后续真实候选集和 worker 行为，所以它更接近离线推荐强化学习实验，而不是完整在线多智能体调度系统。

## 6. 成员 E

### 6.1 可视化训练与多变体对比

成员 E 负责将训练过程转化为可分析的图像结果。`visualize_train.py` 会分别训练 worker 目标和 requester 目标，并保存训练曲线、验证曲线和测试集柱状图；`plot_from_csv.py` 可以从已有 `metrics.csv` 重新生成图像，方便在报告中展示。

成员 E 使用统一实验入口对 DQN 系列变体进行对比。对比时固定数据集划分、训练 epoch、学习率和随机种子，只改变模型变体：

```text
dqn
double_dqn
dueling_dqn
```

对比指标包括：

| 指标                             | 含义                           |
| -------------------------------- | ------------------------------ |
| `worker_test.eval_hit_rate`    | 参与者目标下测试集命中率       |
| `requester_test.eval_hit_rate` | 请求者目标下测试集命中率       |
| `worker_val_best_hit`          | worker 目标验证集最佳命中率    |
| `requester_val_best_hit`       | requester 目标验证集最佳命中率 |
| `loss`                         | 训练稳定性                     |

基础 DQN 直接用 target network 计算最大下一步 Q 值；Double DQN 分离动作选择和动作估值，减少 Q 值过估计；Dueling DQN 将状态价值和动作优势拆开建模，适合分析不同候选任务在同一状态下的相对优势。

最终，成员 E 通过训练曲线、测试集柱状图和多变体汇总表，分析 requester 目标与 worker 目标之间的差异：worker 目标更关注是否命中个人兴趣，requester 目标则进一步强调高质量 worker、任务紧迫度和任务完成价值。
